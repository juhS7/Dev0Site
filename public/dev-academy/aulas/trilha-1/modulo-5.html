🎯 Objetivo

Modelar dados relacionais, escrever consultas SQL profissionais (JOINs, subqueries, CTEs), entender índices e transações, e aproveitar recursos avançados específicos do PostgreSQL.

1. Modelagem relacional

Um banco relacional organiza dados em tabelas (relações) compostas por colunas (atributos) e linhas (registros). Cada tabela tem uma chave primária que identifica unicamente uma linha. Chaves estrangeiras ligam tabelas entre si — esse é o "relacional" do nome.

Antes de escrever SQL, desenhe seu modelo: que entidades existem, quais campos cada uma tem, e como elas se relacionam (1:1, 1:N ou N:M). Para N:M você sempre precisa de uma tabela de junção (ex: user_roles).

2. DDL — criando estruturas

CREATE TABLE users (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email      TEXT UNIQUE NOT NULL,
  nome       TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE posts (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  titulo     TEXT NOT NULL,
  conteudo   TEXT,
  publicado  BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE posts ADD COLUMN views INT NOT NULL DEFAULT 0;

3. DML — manipulando dados

INSERT INTO users (email, nome) VALUES ('ana@x.com', 'Ana');

SELECT id, nome FROM users WHERE nome ILIKE 'a%' ORDER BY created_at DESC LIMIT 10;

UPDATE posts SET publicado = true WHERE id = '...';

DELETE FROM posts WHERE created_at < now() - interval '1 year';

4. JOINs

JOINs combinam linhas de tabelas relacionadas. Os quatro principais:

  • INNER JOIN: só linhas com match em ambos os lados.
  • LEFT JOIN: todas as linhas da esquerda, NULL onde não há match.
  • RIGHT JOIN: espelho do anterior.
  • FULL JOIN: união de ambos, com NULL onde não há match.
-- Posts com nome do autor
SELECT p.titulo, u.nome AS autor
FROM posts p
INNER JOIN users u ON u.id = p.user_id
WHERE p.publicado = true
ORDER BY p.created_at DESC;

-- Usuários e contagem de posts (mesmo quem não tem post)
SELECT u.nome, COUNT(p.id) AS total_posts
FROM users u
LEFT JOIN posts p ON p.user_id = u.id
GROUP BY u.id, u.nome;
💡 Dica

Quando ler "para cada X, mesmo se não tiver Y", é LEFT JOIN. Esse padrão aparece em quase todo dashboard.

5. Subqueries e CTEs (WITH)

CTEs (Common Table Expressions) deixam queries complexas legíveis quebrando em etapas nomeadas.

WITH top_autores AS (
  SELECT user_id, COUNT(*) AS total
  FROM posts
  WHERE publicado = true
  GROUP BY user_id
  HAVING COUNT(*) > 10
)
SELECT u.nome, t.total
FROM top_autores t
JOIN users u ON u.id = t.user_id
ORDER BY t.total DESC;

6. Índices e performance

Um índice é uma estrutura paralela (geralmente B-Tree) que acelera buscas em colunas específicas — em troca de espaço e de escrita um pouco mais lenta.

CREATE INDEX idx_posts_user_id   ON posts(user_id);
CREATE INDEX idx_posts_created   ON posts(created_at DESC);
CREATE INDEX idx_users_email     ON users(email);

EXPLAIN ANALYZE
SELECT * FROM posts WHERE user_id = '...' ORDER BY created_at DESC LIMIT 20;
⚠ Atenção

Não saia criando índice em toda coluna. Cada índice custa espaço e tempo em INSERT/UPDATE. Crie depois de medir com EXPLAIN ANALYZE e só onde queries reais precisam.

7. Transações e ACID

Uma transação agrupa operações que devem ocorrer "tudo ou nada" (atomicidade). PostgreSQL respeita ACID por padrão.

BEGIN;
UPDATE conta SET saldo = saldo - 100 WHERE id = 1;
UPDATE conta SET saldo = saldo + 100 WHERE id = 2;
-- se algo falhar:
ROLLBACK;
-- se tudo deu certo:
COMMIT;

8. PostgreSQL especificamente

Postgres tem recursos que você usaria todo dia em projetos sérios:

  • JSONB: armazena JSON binário com índices GIN — bom para campos semi-estruturados.
  • Arrays nativos: tags TEXT[] sem precisar de tabela auxiliar.
  • UUID como chave primária via gen_random_uuid().
  • Full-text search embutido com tsvector/tsquery.
CREATE TABLE eventos (
  id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tipo    TEXT NOT NULL,
  payload JSONB NOT NULL,
  tags    TEXT[] DEFAULT '{}',
  ts      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_eventos_payload ON eventos USING GIN (payload);
CREATE INDEX idx_eventos_tags    ON eventos USING GIN (tags);

-- consultar dentro do JSONB
SELECT * FROM eventos
WHERE payload ->> 'user_id' = 'u_123'
  AND 'erro' = ANY(tags);
